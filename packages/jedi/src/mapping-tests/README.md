# Mapping Test Fixtures

Test fixtures validate that JSONata mapping expressions produce the correct TMS output for real production EDI files.

## Directory Structure

Each mapping has its own directory, with subdirectories for individual fixtures:

```
fixtures/
  expeditors-204-inbound/          ← mapping slug
    NWKD_204/                       ← fixture name (from uploaded filename)
      input.edi                     ← raw X12 EDI file
      expected-output.json          ← expected TMS JSON output
      mapping.jsonata               ← JSONata mapping expression snapshot
      meta.json                     ← { "source": "stedi"|"generated", "uploadedAt": "..." }
    shipment-002/
      input.edi
      expected-output.json
      mapping.jsonata
      meta.json
```

All three core files (input.edi, expected-output.json, mapping.jsonata) are required per fixture. meta.json is created automatically on upload.

## Fixture Sources

- **`source: "stedi"`** — Ground truth. The expected-output.json was uploaded directly from Stedi's actual output. Eddie's mapping must match this exactly.
- **`source: "generated"`** — Auto-generated by running Eddie's mapping. Used for regression testing — if the output changes unexpectedly, the test fails.

## Adding Fixtures

### 1. Paired Upload — Ground Truth (recommended for migration)
Upload two files together via the UI or API:
- `NWKD_211.edi` — the raw carrier EDI file
- `NWKD_211.json` — the EXACT output Stedi produces today

The .json file becomes the ground truth expected-output.json. The fixture gets `source: "stedi"` in meta.json.

Naming convention: `{TRADINGPARTNERCODE}_{TRANSACTIONSET}.edi` + matching `.json`
Examples: `FWDA_211.edi` + `FWDA_211.json`, `NWKD_204.edi` + `NWKD_204.json`

### 2. Single .edi Upload — Auto-generated
Upload just the .edi file. Eddie runs the mapping and saves the output as expected-output.json. Gets `source: "generated"`.

### 3. Codebase Drop
Create a directory in `fixtures/{mapping-slug}/{fixture-name}/` with all required files.

## Running Tests

```bash
# Run all mapping fixture tests
pnpm test:mappings

# Filter by carrier name
pnpm test:mappings --carrier=Expeditors

# Update expected-output.json snapshots from current mapping output
pnpm test:mappings --update
```

## How It Works

Each test fixture runs through this pipeline:
1. **Parse**: `input.edi` → x12Parser → ParsedEnvelope
2. **Transform**: ParsedEnvelope → toJedi() → JediDocument (auto-routes by transaction set)
3. **Map**: JediDocument + `mapping.jsonata` → JsonataEvaluator → TMS output
4. **Compare**: TMS output vs `expected-output.json` (deep equality)
5. **Validate**: TMS output vs defaultTmsSchema (required fields check)

A fixture **passes** when both comparison and validation succeed.

For `source: "stedi"` fixtures, comparison against expected-output.json is strict — this IS what the TMS expects.

For `source: "generated"` fixtures, comparison is a regression guard — output must not change unexpectedly.

## Updating Snapshots

When you intentionally change a mapping expression, run:

```bash
pnpm test:mappings --update
```

This re-evaluates every fixture's mapping and overwrites `expected-output.json` with the new output. **Only affects `source: "generated"` fixtures.** Stedi ground truth fixtures should not be updated — if they fail, the mapping needs fixing, not the expected output.
